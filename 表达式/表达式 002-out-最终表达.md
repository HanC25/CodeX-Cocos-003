# 步骤二：开场细节调整最终表达

## 最终目标

在步骤一已经完成并验证的“游戏开始画面1”基础上，对 yu1、yu2、yu3、yu4 的缩放、掉落轨道、掉落节奏、掉落摆动、停止动作和停止后的浮动进行调整。

本步骤最终实现时，继续保留步骤一已经确认的基础规则：

1. 游戏使用 Cocos Creator 3.8.8。
2. 画布为横屏 1280x720。
3. 背景使用 `assets/art/common/textures/群组111.jpg`，并铺满整个 1280x720 游戏画布。
4. yu1、yu2、yu3、yu4 使用各自对应的 Spine 资源。
5. 所有位置判断都以缩放后的 Spine 可见内容为准，而不是以节点原点为准。
6. Spine 透明显示必须正常，不能出现白底、彩边、黑边或透明异常。
7. 场景中不能出现 missing script 或脚本挂载错误。

## 对象定义

yu1 -> `assets/art/common/spine/11`

yu2 -> `assets/art/common/spine/22`

yu3 -> `assets/art/common/spine/33`

yu4 -> `assets/art/common/spine/44`

## 最终表现

1. yu1、yu2、yu3、yu4 的显示大小统一设置为步骤一基础大小的 80%。
   缩放值直接设置为 0.8，不在当前缩放基础上继续叠乘。

2. 1280x720 画布在水平方向平均分为 4 个等宽区域。
   从左到右，4 个区域的中轴线分别定义为 z1、z2、z3、z4。
   在 Cocos Canvas 以画布中心为原点的坐标系中：

   z1 = -480

   z2 = -160

   z3 = 160

   z4 = 480

3. yu1、yu2、yu3、yu4 分别沿 z1、z2、z3、z4 掉落。
   掉落轨道以缩放后的 Spine 可见内容视觉中心对齐对应中轴线为准。

4. 开场初始状态下，yu1、yu2、yu3、yu4 的缩放后可见内容必须整体位于屏幕上方之外。
   不能只让节点原点位于屏幕外。

5. yu1、yu2、yu3、yu4 的掉落顺序每次进入画面时重新随机。
   随机顺序只影响开始掉落的先后，不改变 yu1-yu4 各自对应的 z1-z4 掉落轨道。

6. yu1、yu2、yu3、yu4 不是同时开始掉落。
   相邻两个物体开始掉落的时间间隔，每次在 0.3 秒到 0.6 秒之间独立随机。
   随机出的间隔在本轮开场动画中保持固定。

7. 每个对象的掉落时间在 3 秒到 6 秒之间独立随机。
   随机出的掉落时间在该对象本次掉落过程中保持固定。

8. 每个对象掉落过程中播放 Spine 动作：

   `1_zhangzui`

   该动作在掉落期间循环播放。

9. 每个对象掉落过程中，以自身对应的 z 轨道为中心进行左右摆动。
   左右摆动幅度在 5px 到 30px 之间独立随机。
   左右摆动次数按整数随机，只允许为 3、4、5、6 次完整左右摆动循环。
   摆动幅度和摆动次数在该对象本次掉落过程中保持固定。

10. 每个对象到达停止位置后，切换播放 Spine 动作：

    `6_jingzhi`

    该动作在停止和浮动期间循环播放。

11. 每个对象的最终停止位置继承步骤一规则。
    停止时，缩放后的 Spine 可见底边距离屏幕底部 20px。
    不能用节点原点距离屏幕底部 20px 来判断。

12. 每个对象到达停止位置后，立即进入上下浮动状态，不需要等待其他对象全部掉落完成。
    上下浮动以该对象的最终停止位置为中心。
    上下浮动幅度在 5px 到 20px 之间独立随机。
    上下浮动频次在 2 次 / 3 秒到 3 次 / 3 秒之间独立随机。
    浮动幅度和浮动频次在该对象停止后的浮动过程中保持固定。

## 实现确认

1. `GameStart.ts` 中使用随机范围字段控制开场表现：
   `minStartInterval`、`maxStartInterval`、`minDropDuration`、`maxDropDuration`、`minSwingAmplitude`、`maxSwingAmplitude`、`minSwingCycles`、`maxSwingCycles`、`minFloatAmplitude`、`maxFloatAmplitude`、`minFloatCyclesPerDuration`、`maxFloatCyclesPerDuration`。

2. `main.scene` 中的 GameStart 组件序列化字段已经同步为上述随机范围字段。
   旧的固定字段不再保留：
   `startInterval`、`dropDuration`、`swingAmplitude`、`swingCycles`、`floatAmplitude`、`floatCyclesPerDuration`、`floatDuration`。

3. `main.scene` 中 GameStart 脚本组件使用 Cocos Creator 可识别的脚本类型序列化 ID，不能写成普通类名，避免出现 missing script。

4. 4 个 Spine 组件的 `_premultipliedAlpha` 保持为 `false`，保证透明显示正常。

1. yu1-yu4 的显示大小为步骤一基础大小的 80%。
2. yu1-yu4 分别沿 z1、z2、z3、z4 掉落。
3. 掉落轨道以缩放后的 Spine 可见内容视觉中心判断。
4. 开场初始状态下，yu1-yu4 的缩放后可见内容完全位于屏幕上方之外。
5. yu1-yu4 的掉落顺序每次进入画面时随机。
6. 相邻两个物体开始掉落的间隔随机在 0.3 秒到 0.6 秒之间。
7. 每个物体从开始掉落到停止的时间随机在 3 秒到 6 秒之间。
8. 掉落过程中，每个正在掉落的 Spine 循环播放 `1_zhangzui`。
9. 掉落过程中，每个物体左右摆动幅度随机在 5px 到 30px 之间。
10. 掉落过程中，每个物体左右摆动次数为 3、4、5、6 次中的一个随机整数。
11. 每个物体到达停止位置后，切换循环播放 `6_jingzhi`。
12. 停止时，每个 Spine 的缩放后可见底边距离屏幕底部 20px。
13. 停止后，每个物体立即以停止位置为中心上下浮动。

15. 上下浮动频次随机在 2 次 / 3 秒到 3 次 / 3 秒之间。
16. Spine 透明正常，没有白底、彩边、黑边。
17. Cocos Creator 控制台没有 missing script 或脚本挂载错误。
18. TypeScript 编译通过。
19. `main.scene` JSON 解析通过。

## 最终表达与规范表达的区别

1. `规范表达` 是编码前的执行说明，重点是把原始需求补充成可编码、可验收、可规避风险的规则。
2. `最终表达` 是实现成功并验证后的稳定沉淀，重点是记录已经被证明可行的最终口径。
3. `规范表达` 可以包含“如果不存在动画名，需要实现前确认”等预防性说明。
4. `最终表达` 应写成确定结果，例如本步骤最终确认停止动作为 `6_jingzhi`。
5. `规范表达` 面向“准备实现”，会强调风险、歧义和实现前检查。
6. `最终表达` 面向“后续复用”，会记录最终采用的字段、脚本序列化、透明设置和验证结果。
7. `规范表达` 允许保留待确认项。
8. `最终表达` 不应保留待确认项，除非该问题确实未解决。
