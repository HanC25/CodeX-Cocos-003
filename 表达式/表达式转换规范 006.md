# 表达式转换规范 006

## 目标

本规范基于 `表达式转换规范 005.md`，并吸收 `表达式 006` 系列在设置界面、弹窗遮罩、按钮分组、临时设置、确认保存、入口规则覆盖、数字转盘和裁剪容器中的经验。

用于指导后续把 `原始表达` 转换为 `规范表达`，再从实现结果沉淀为 `最终表达`。

本版本新增重点：

1. 设置界面必须区分显示层、遮罩层、按钮层和数据状态。
2. 弹窗类 UI 必须明确是否阻止下方内容点击。
3. 单选按钮组必须明确默认值、互斥规则、选中资源和未选中资源。
4. 设置参数必须明确“临时设置”和“已保存设置”的关系。
5. 关闭与确认必须分别定义：关闭丢弃，确认保存。
6. 后续步骤覆盖前置步骤入口规则时，必须显式写出覆盖关系。
7. 数字转盘类控件必须明确显示范围、循环规则、吸附规则和拖动手感。
8. “不能移出区域”不应优先用硬性位移截断实现，应优先用裁剪容器实现。
9. 最终表达需要记录真实节点父子关系、脚本字段、关键方法和本地预览结果。

## 三类表达的职责

### 原始表达

`原始表达` 记录用户最初输入和后续追加的修改说明。

要求：

1. 保留用户原句。
2. 保留用户写出的资源路径、对象名和参考图说明。
3. 保留用户对默认值、点击交互、滑动交互和关闭确认行为的原始描述。
4. 用户追加修改说明时，需要追加到同一个步骤的原始表达中。
5. 即使原始表达中出现命名不一致，也需要保留原句，并在规范表达中统一。

### 规范表达

`规范表达` 是编码前和修正过程中的执行依据。

要求：

1. 保留原始表达的有效意图。
2. 把设置参数、UI 对象、资源、层级、显示隐藏、点击交互、滑动交互和验收标准拆成独立规则。
3. 明确默认值、可选值和数据类型。
4. 明确按钮组互斥规则。
5. 明确设置界面显示时是否阻止下方内容点击。
6. 明确关闭和确认对设置参数的不同影响。
7. 明确前置步骤入口规则是否被当前步骤覆盖。
8. 对含糊对象名进行统一，例如 `setconfirm` 统一为 `setok`。

### 最终表达

`最终表达` 是实现成功并验证后的稳定沉淀。

要求：

1. 只记录最终确认采用的规则。
2. 记录最终采用的资源路径、节点结构和脚本字段。
3. 记录实现中修正过的问题和稳定解决方案。
4. 记录测试验证结果。
5. 能作为后续步骤的可靠前置基础。

## 设置参数规则

如果需求出现设置界面、选项、默认值或参数保存，规范表达必须明确：

1. 参数名。
2. 参数类型。
3. 可选值范围。
4. 默认值。
5. 显示含义。
6. 修改时机。
7. 保存时机。
8. 关闭时是否丢弃修改。

示例：

```text
gameperson 可选值为 single、double，默认 single。
打开设置界面时，从 savedSettings 复制 tempSettings。
点击按钮时，只修改 tempSettings。
点击 setclose 时，丢弃 tempSettings。
点击 setok 时，将 tempSettings 写入 savedSettings。
```

## 弹窗遮罩规则

如果需求出现弹窗、设置界面、菜单界面或背景变暗，规范表达必须明确：

1. 是否需要全屏遮罩。
2. 遮罩是否显示在背景和面板之间。
3. 遮罩是否阻止下层点击。
4. 面板是否显示在遮罩之上。
5. 弹窗关闭后遮罩是否隐藏。

示例：

```text
设置界面显示时，setmask 全屏显示，背景变暗。
setmask 阻止下方封面或游戏内容点击。
设置界面关闭后，setmask 和所有设置界面对象隐藏。
```

## 入口规则覆盖

如果当前步骤改变了前一步的开始入口，规范表达必须明确写出覆盖关系。

需要说明：

1. 哪个前置步骤的规则被覆盖。
2. 哪些旧入口不再触发原行为。
3. 哪个新入口触发新行为。
4. 主流程最终从哪个动作开始。

示例：

```text
步骤六覆盖步骤五的封面入口规则。
fmlab01、fmlab02、fmclose 不开始游戏。
fmstart 只打开设置界面。
setok 保存设置后才开始游戏。
```

## 按钮组规则

如果需求出现两个或多个互斥选项，规范表达必须形成按钮组规则。

每组必须明确：

1. 组名。
2. 包含哪些按钮。
3. 默认选中项。
4. 任意时刻是否必须有且只有一个选中。
5. 每个按钮选中时写入的参数值。
6. 每个按钮选中和未选中资源。

示例：

```text
setfast 和 setnormal 是游戏速度按钮组。
默认选中 setnormal。
点击 setfast 后，gamespeed = fast。
点击 setnormal 后，gamespeed = normal。
该组任意时刻有且只有一个按钮处于选中状态。
```

## 关闭和确认规则

设置界面、菜单界面、弹窗界面必须区分关闭和确认。

规范表达必须明确：

1. 关闭按钮对象名。
2. 确认按钮对象名。
3. 关闭是否保存参数。
4. 确认是否保存参数。
5. 关闭后返回哪个界面。
6. 确认后是否进入主流程。

示例：

```text
点击 setclose，关闭设置界面，不保存本次临时修改。
点击 setok，保存当前设置参数，关闭设置界面，并开始游戏。
```

## 数字转盘规则

如果需求出现数字转盘、滑动选择、循环数字或吸附效果，规范表达必须明确：

1. 可选数字范围。
2. 默认选中数字。
3. 同时显示几个数字。
4. 哪个位置代表当前选中值。
5. 左右滑动分别对应增加还是减少。
6. 是否循环显示。
7. 拖动过程中数字是否跟手。
8. 松手后是否吸附。
9. 是否允许半个数字停留。
10. 数字是否允许移出显示区域。
11. 显示区域越界的实现原则。

示例：

```text
gametime 可选值为 1-9，默认 3。
时间转盘显示 3 个数字，中间数字为当前选中值。
向左滑动时，当前选中数字增加。
向右滑动时，当前选中数字减少。
松手后必须吸附到完整数字位置。
不能出现半个数字停留。
```

## 裁剪优先规则

当需求要求“内容不能移出显示区域”时，规范表达不能只写“限制位移”。

需要优先判断是否应该使用裁剪容器。

适用场景：

1. 数字转盘。
2. 横向列表。
3. 滚动选择器。
4. 跑马灯。
5. 任何需要连续跟手移动但不能显示出界的 UI。

规范表达应写为：

```text
内容需要连续跟随手指移动。
显示范围通过父容器裁剪控制。
不能通过硬性截断位移导致拖动卡顿或突然跳变。
```

最终表达需要记录：

1. 哪个节点作为裁剪容器。
2. 哪些节点作为裁剪容器子节点。
3. 是否使用 Mask 或同等裁剪机制。
4. 子节点本地坐标。

## 参考图和底图规则

如果参考图和实际底图已经包含某些视觉元素，规范表达必须明确不重复叠加。

需要检查：

1. 背景底图是否已经包含浅色条。
2. 底图是否已经包含选中框。
3. 是否还需要独立切图。
4. 哪些对象只显示动态内容。

示例：

```text
cd12.png 已经包含时间设置区浅色长条和中间数字选中框。
settime 不额外显示背景图。
settimeLeft、settimeCenter、settimeRight 只显示数字。
```

## 对象命名统一规则

如果原始表达中同一对象出现多个名称，规范表达必须统一正式对象名。

要求：

1. 保留原始名称来源。
2. 明确正式对象名。
3. 后续代码和最终表达只使用正式对象名。

示例：

```text
原始表达第 8 条中的 setconfirm 视为 setok 的同义描述，正式对象名使用 setok。
```

## 脚本字段规则

新增设置界面或复杂 UI 时，规范表达必须要求脚本字段与场景序列化字段一致。

需要明确：

1. 哪些节点属于统一显示隐藏集合。
2. 哪些节点属于入口集合。
3. 哪些节点属于按钮组。
4. 哪些节点属于触摸区域集合。
5. 哪些资源是选中态。
6. 哪些资源是未选中态。

示例：

```text
settingNodes = [setmask, setpanel, setdouble, setsingle, setfast, setnormal, setclose, setok, settime, settimeLeft, settimeCenter, settimeRight]
settingOpenNodes = [fmstart]
settimeTouchNodes = [settime, settimeLeft, settimeCenter, settimeRight]
```

## 本地预览测试规则

设置界面、菜单界面和弹窗完成后，至少需要验证：

1. 首屏是否仍符合前置流程。
2. 入口按钮是否打开弹窗或菜单。
3. 弹窗或菜单是否显示在正确层级。
4. 遮罩是否阻止下方点击。
5. 默认选项是否正确。
6. 按钮组互斥是否正确。
7. 关闭是否丢弃临时修改。
8. 确认是否保存并进入正确流程。
9. 滑动控件是否跟手。
10. 滑动控件松手后是否吸附。
11. 控制台是否有 error。
12. 是否出现 missing script 或脚本挂载错误。

## 验收标准生成规则

设置界面类验收标准按以下顺序组织：

1. 首屏和入口规则。
2. 弹窗显示规则。
3. 遮罩和点击阻断规则。
4. 参数默认值。
5. 按钮组互斥规则。
6. 设置关闭和确认规则。
7. 数字转盘显示规则。
8. 数字转盘拖动和吸附规则。
9. 资源重复叠加检查。
10. 控制台、scene JSON、资源 UUID、脚本挂载校验。
11. 本地预览测试结果。

## 最终表达沉淀规则

从 `规范表达` 生成 `最终表达` 时，需要：

1. 写清最终采用的入口规则。
2. 写清前置步骤中被覆盖的规则。
3. 写清最终采用的资源路径。
4. 写清最终采用的节点层级。
5. 写清最终采用的脚本字段和关键方法。
6. 写清临时设置与已保存设置的关系。
7. 写清复杂控件的最终实现原则。
8. 写清本地预览验证结果。
9. 避免保留“需要确认”“例如”等未完成语气。

## 写作风格

1. 使用明确、直接的中文。
2. 一个条目只表达一个判断点。
3. 对路径、节点名、方法名和字段名使用反引号。
4. 对入口、遮罩、按钮组、参数状态、复杂控件分别成章描述。
5. 发现命名不一致时，必须显式写出统一结论。
6. 发现视觉资源已经包含某个元素时，必须写出不要重复叠加。
