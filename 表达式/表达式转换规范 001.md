# 表达式转换规范

## 目标

将用户输入的 `原始表达`，转换为可直接指导编码实现的 `规范表达`。
在实现并修正问题后，再沉淀为 `最终表达`。

转换后的 `规范表达` 必须做到：

1. 保留用户原始意图。
2. 补足实现时必须明确的技术口径。
3. 把容易产生歧义的描述改写成可执行规则。
4. 把隐藏风险提前暴露为约束。
5. 增加明确的验收标准。

## 总体结构

`规范表达` 和 `最终表达` 使用以下结构：

```text
<功能/画面标题>：

<项目基础信息>
<资源与对象定义>

<行为/表现分组标题>：

1. <核心表现>
2. <关键约束解释>
3. <实现口径>
...

验收标准：

1. <可观察结果>
2. <可验证技术结果>
...
```

## 三类表达

每一个步骤固定保留 3 个表述：

1. `表达式 00X-in-原始表达.md`
   记录用户最初的需求描述。

2. `表达式 00X-in-规范表达.md`
   记录 Codex 协助用户整理后的规范需求描述，用于正式编码前确认。

3. `表达式 00X-out-最终表达.md`
   记录修正完 bug、验证完效果后，重新提炼出的最终需求描述。

统一术语：

1. `原始表达`：用户最初输入的自然语言需求。
2. `规范表达`：编码前整理出的、可执行、可验证、尽量规避风险的需求描述。
3. `最终表达`：实现并修正问题后沉淀的稳定表达，用于后续复用和对照。

## 转换规则

### 1. 去除代码块包裹

如果 `原始表达` 整体被 Markdown 代码块包裹：

```text
```text
...
```
```

输出时去掉外层代码块，正文直接作为 Markdown 文档。

### 2. 标题保持原意

保留输入中的主标题，例如：

```text
游戏开始画面1：
```

不要改成抽象标题。标题代表功能模块、画面阶段或行为阶段。

### 3. 项目信息改写为完整句

输入：

```text
项目为 Cocos Creator 3.8.8，横屏设计分辨率 1280x720。
```

输出：

```text
项目为 Cocos Creator 3.8.8，横屏游戏，设计分辨率为 1280x720。
```

要求：

1. 明确引擎版本。
2. 明确横屏/竖屏。
3. 明确设计分辨率。
4. 分辨率写成 `宽x高`。

### 4. 背景资源必须说明铺满范围

输入：

```text
背景使用 assets/art/common/textures/群组111.jpg，铺满画布。
```

输出：

```text
背景使用 assets/art/common/textures/群组111.jpg，铺满整个 1280x720 游戏画布。
```

要求：

1. 保留资源路径。
2. 明确铺满的是整个游戏画布。
3. 如果已知设计分辨率，写出具体尺寸。

### 5. 对象列表保持映射关系

输入：

```text
yu1 -> assets/art/common/spine/11
yu2 -> assets/art/common/spine/22
```

输出保持同样映射，但上下增加空行，使其成为独立对象定义区：

```text
场景中有 4 个 Spine 物体：

yu1 -> assets/art/common/spine/11
yu2 -> assets/art/common/spine/22
```

要求：

1. 对象名不能改。
2. 资源路径不能改。
3. 映射关系不能合并成自然语言，以免丢失精确性。

### 6. 将“要求”改写成表现分组

输入中的：

```text
要求：
```

应根据语义改写成更具体的分组标题。

例如游戏开场行为改成：

```text
开场表现：
```

如果是 UI 行为，可改成：

```text
交互表现：
```

如果是结算逻辑，可改成：

```text
结算规则：
```

### 7. 简短要求必须扩展成可编码规则

输入中的一句话要求，输出时不能只换一种说法，必须展开为：

1. 表现目标。
2. 不能怎么理解。
3. 正确实现口径。

例如：

输入：

```text
4 个物体按“可见内容的视觉中心”在屏幕宽度内等间距排布，而不是按节点原点等间距。
```

输出：

```text
4 个 Spine 在水平方向需要等间距排布。
等间距的计算标准不是节点原点，而是 Spine 可见内容的视觉中心。
也就是说，需要根据每个 Spine 的 skeleton bounds 计算可见区域中心点，
再让 4 个可见中心在 1280 宽度内等间距分布。
```

### 8. 涉及坐标时必须说明参照物

凡是出现以下词语：

```text
上方
底部
中心
等间距
停止位置
距离
屏幕外
```

输出必须补充参照物。

示例：

```text
不能只让节点原点在屏幕外，而是要求 Spine 实际可见图像完全不出现在画面中。
```

```text
停止后，每个 Spine 的可见底边距离屏幕底部 20 像素。
不能用节点原点距离底部 20 像素来判断。
```

### 9. Spine 对象必须补充可见区域口径

如果对象类型是 Spine，输出必须补充以下风险约束：

1. 节点原点不等于视觉中心。
2. 节点原点不等于可见底边。
3. 应根据 skeleton bounds 判断可见内容。
4. 如果涉及透明，必须检查 PNG alpha 和 premultipliedAlpha。

### 10. 动画要求必须拆成动作名和播放方式

输入：

```text
每个 Spine 播放 1_zhangzui 动作，循环播放。
```

输出：

```text
掉落过程中，4 个 Spine 同时播放动作：
1_zhangzui
该动作循环播放。
```

要求：

1. 动作名独占一行。
2. 明确是否循环。
3. 明确播放时机。
4. 明确是同时播放还是分别播放。

### 11. 透明显示必须作为独立风险项

输入如果提到透明、Spine、PNG、白底、彩边、alpha，输出必须加入独立条目：

```text
Spine 透明显示必须正确。
需要检查每个 Spine PNG 是否带 alpha 通道。
如果资源是普通 straight alpha，Spine 组件需要关闭 premultipliedAlpha。
最终画面不能出现白底、彩边、黑边或透明区域异常。
```

### 12. Cocos 脚本挂载必须作为独立风险项

如果需求涉及新增脚本、场景逻辑、自动播放、动画控制，输出必须加入：

```text
脚本必须正确挂载到场景中。
打开 Cocos Creator 后不能出现：
missing script
missing or invalid script
Script attached to Canvas is missing or invalid
等错误。
如果直接修改 .scene 文件，需要使用 Cocos Creator 正确的脚本组件序列化方式，
不能简单把脚本类名写成 __type__。
```

## 验收标准生成规则

`out.md` 必须包含 `验收标准：`。

验收标准按以下顺序组织：

1. 静态画面是否正确。
2. 初始状态是否正确。
3. 动态表现是否正确。
4. 最终状态是否正确。
5. 资源显示是否正确。
6. 编辑器/控制台是否无错误。
7. 技术校验是否通过。

示例：

```text
验收标准：

1. 打开场景时，背景铺满 1280x720。
2. 游戏开始前，yu1-yu4 完全在屏幕上方之外，画面内不可见。
3. 游戏开始后，yu1-yu4 同时从屏幕上方掉落。
4. 掉落过程中，4 个 Spine 都播放 1_zhangzui 动作。
5. 停止后，4 个 Spine 的可见底边都距离屏幕底部 20px。
6. 停止后，4 个 Spine 的可见内容视觉中心在水平方向等间距。
7. Spine 透明正常，没有白底、彩边、黑边。
8. Cocos Creator 控制台没有 missing script 或脚本挂载错误。
9. scene JSON、资源 UUID、脚本类型检查都需要通过。
```

## 写作风格

1. 使用明确、直接的中文。
2. 每条规则只表达一个判断点。
3. 避免“看起来”“差不多”“适当”等模糊词。
4. 保留英文技术词，例如 `Spine`、`skeleton bounds`、`premultipliedAlpha`、`missing script`。
5. 路径、动作名、错误文本必须原样保留。
6. 不使用代码块包裹整篇输出。

## 转换检查清单

生成 `规范表达` 或 `最终表达` 前，检查是否回答了这些问题：

1. 谁是对象？
2. 对象对应哪些资源？
3. 初始状态是什么？
4. 过程中发生什么？
5. 最终状态是什么？
6. 坐标和距离以什么为参照？
7. 是否存在资源显示风险？
8. 是否存在脚本挂载风险？
9. 如何判断实现合格？
