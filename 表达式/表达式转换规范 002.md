# 表达式转换规范 002

## 目标

本规范基于 `表达式转换规范 001.md`，并吸收 `表达式 002` 系列在开场细节调整中的经验，用于指导后续把 `原始表达` 转换为 `规范表达`，再从实现结果沉淀为 `最终表达`。

本版本新增重点：

1. 多步骤需求需要明确继承关系。
2. 随机行为必须说明随机范围、随机时机和随机后是否保持固定。
3. 动画名称必须以资源实际存在的名称为准。
4. Cocos 场景脚本字段必须和 TypeScript 属性同步。
5. 最终表达必须记录实现中已经验证过的稳定技术口径。

## 三类表达的职责

### 原始表达

`原始表达` 记录用户最初的自然语言需求。

要求：

1. 尽量保留用户原句。
2. 只做必要的错别字或动作名同步修正。
3. 不主动加入大量工程解释。
4. 如果用户后续修改了需求，需要同步更新原始表达。

### 规范表达

`规范表达` 是正式编码前的执行依据。

要求：

1. 保留原始表达的全部有效意图。
2. 补充工程实现口径。
3. 明确坐标、尺寸、可见内容、动画、随机、资源、脚本挂载等风险点。
4. 把模糊描述改写为可执行规则。
5. 增加验收标准。
6. 允许包含实现前需要确认的风险提示。

### 最终表达

`最终表达` 是实现成功并验证后的稳定沉淀。

要求：

1. 记录最终确认采用的需求口径。
2. 删除已经解决的待确认表述。
3. 保留实现中证明重要的技术约束。
4. 记录最终采用的关键字段、动作名、资源设置和验证结果。
5. 作为后续步骤的可靠前置基础。

## 总体结构

`规范表达` 推荐结构：

```text
<步骤标题>

目标：
<说明本步骤要调整什么>

基础约束：
<继承哪些已确认规则>

对象定义：
<对象与资源映射>

表现规则：
<逐条说明行为>

验收标准：
<逐条列出可观察和可验证结果>
```

`最终表达` 推荐结构：

```text
<步骤标题>最终表达

最终目标
<已经验证通过的目标>

对象定义
<对象与资源映射>

最终表现
<最终采用的表现规则>

实现确认
<实现中已经确认的重要技术口径>

验收标准
<最终可复验结果>

最终表达与规范表达的区别
<说明本步骤沉淀出的差异>
```

## 继承关系规则

如果原始表达出现以下说法：

```text
调整步骤 1 的一些细节
在上一步基础上
继续修改
保持之前规则
```

规范表达必须明确：

1. 当前步骤继承哪些已完成规则。
2. 当前步骤只修改哪些规则。
3. 当前步骤不能破坏哪些已验证结果。
4. 当前步骤与前一步冲突时，以当前步骤的明确修改为准。

示例：

```text
在步骤一已完成的“游戏开始画面1”基础上进行调整。
不破坏步骤一已经确认的背景、资源映射、可见内容定位、透明显示和脚本挂载规则。
```

## 随机行为转换规则

凡是原始表达中出现：

```text
随机
随机在 A～B
A 到 B 之间
每次随机
随机顺序
```

规范表达必须补充以下内容：

1. 随机范围。
2. 随机单位。
3. 随机时机。
4. 是否每个对象独立随机。
5. 随机后是否在本轮动画中保持固定。
6. 随机值是连续值还是整数值。

示例：

```text
每次进入画面时，每个相邻掉落间隔都需要在 0.3 秒到 0.6 秒之间独立随机一次。
随机出的间隔在本轮开场动画中保持固定，不能在动画过程中反复变化。
```

```text
摆动频次按整数次数随机，只允许出现 3、4、5、6 次完整左右摆动循环。
```

## 时间与频次规则

涉及时间、间隔、持续时间、频次时，必须明确：

1. 单位是秒、毫秒还是帧。
2. 间隔指开始时间间隔，还是动作完成后的等待间隔。
3. 持续时间指从哪个时刻到哪个时刻。
4. 频次指完整循环次数，而不是单次上/下或左/右位移。
5. 如果写作 `2 次 / 3 秒`，需要解释为每 3 秒完成 2 次完整循环。

## 轨道与坐标规则

涉及画布等分、轨道、中轴线、视觉中心时，必须明确：

1. 画布尺寸。
2. 坐标系原点。
3. 每条轨道的坐标值。
4. 对齐的是节点原点还是可见内容视觉中心。
5. 如果对象经过缩放，所有坐标判断都基于缩放后的可见内容。

示例：

```text
在 Cocos Canvas 以画布中心为原点的坐标系中，z1-z4 的 x 坐标分别为：
z1 = -480
z2 = -160
z3 = 160
z4 = 480
```

## Spine 可见内容规则

如果对象是 Spine，规范表达和最终表达都必须保留以下口径：

1. 节点原点不等于可见内容视觉中心。
2. 节点原点不等于可见底边。
3. 位置判断必须基于 Spine 可见内容。
4. 缩放后的位置判断必须基于缩放后的可见内容。
5. 停止位置、屏幕外起点、轨道对齐、摆动和浮动都要使用同一套可见内容口径。

## Spine 动画规则

涉及 Spine 动作时，规范表达必须拆成：

1. 播放时机。
2. 动作名。
3. 是否循环。
4. 动作切换条件。
5. 动作名是否已在资源中确认存在。

如果实现过程中确认原始表达中的动作名不准确，最终表达必须写最终采用的动作名。

示例：

```text
掉落过程中循环播放 `1_zhangzui`。
到达停止位置后切换循环播放 `6_jingzhi`。
```

## Cocos 场景与脚本规则

涉及 Cocos Creator 场景脚本时，规范表达必须加入：

1. 不能出现 missing script。
2. 场景 JSON 必须能正常解析。
3. 脚本组件序列化必须使用 Cocos Creator 可识别的脚本类型。
4. TypeScript 中暴露给编辑器的 `@property` 字段，需要与 `.scene` 中序列化字段保持一致。
5. 删除或替换旧字段时，需要同步清理 `.scene` 中的旧序列化字段，避免编辑器继续保留旧参数。

示例：

```text
如果将固定参数改为随机范围字段，需要同步更新 GameStart.ts 和 main.scene。
main.scene 中不应继续保留旧的 startInterval、swingAmplitude、floatAmplitude 等字段。
```

## 透明显示规则

涉及 Spine 透明、白底、彩边、黑边时，规范表达和最终表达都必须保留：

1. 检查图片资源是否带 alpha。
2. 检查 Spine 组件的 premultipliedAlpha 设置。
3. 如果资源为普通 straight alpha，`_premultipliedAlpha` 应关闭。
4. 最终画面不能出现白底、彩边、黑边或透明异常。

## 验收标准生成规则

验收标准按以下顺序组织：

1. 基础继承结果是否仍然正确。
2. 静态位置、缩放、轨道是否正确。
3. 初始状态是否正确。
4. 动态顺序、随机间隔、随机持续时间是否正确。
5. 动画播放和切换是否正确。
6. 停止位置和停止后循环表现是否正确。
7. 透明显示是否正确。
8. 控制台和脚本挂载是否无错误。
9. TypeScript、scene JSON、资源 UUID 等技术校验是否通过。

## 最终表达沉淀规则

从 `规范表达` 生成 `最终表达` 时，需要做以下处理：

1. 把“需要确认”的内容改为“最终确认”的结果。
2. 把实现中真正用到的字段、动作名和资源设置写入最终表达。
3. 把已经证明有效的防错规则保留下来。
4. 把未采用的旧字段、旧动作名、旧参数写入“不能保留”或“已清理”说明。
5. 保留最终验收结果，方便后续步骤复用。

## 写作风格

1. 使用明确、直接的中文。
2. 一个条目只表达一个判断点。
3. 不使用“差不多”“大概”“适当”等模糊词。
4. 对随机、坐标、动画、资源、脚本字段使用确定表述。
5. 关键动作名、资源路径、脚本字段使用代码格式。
