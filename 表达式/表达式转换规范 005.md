# 表达式转换规范 005

## 目标

本规范基于 `表达式转换规范 004.md`，并吸收 `表达式 005` 系列在封面界面、资源纠偏、入口阻塞、视觉按钮和实际点击区域分离、场景层级管理中的经验。

用于指导后续把 `原始表达` 转换为 `规范表达`，再从实现结果沉淀为 `最终表达`。

本版本新增重点：

1. 封面、弹窗、菜单等入口界面必须明确是否阻塞主流程。
2. UI 资源路径必须校验实际存在的文件格式和 SpriteFrame UUID。
3. 视觉按钮和真实可点击对象必须分开描述。
4. UI 层级必须明确父节点、子节点顺序和遮挡关系。
5. 点击开始后，需要明确隐藏哪些 UI、显示哪些游戏对象、启动哪个主流程方法。
6. 最终表达需要记录本地预览的首屏、点击入口和控制台验证结果。

## 三类表达的职责

### 原始表达

`原始表达` 记录用户最初输入和后续追加的修改说明。

要求：

1. 保留用户原句。
2. 保留用户写出的资源路径，即使路径后续发现需要纠偏。
3. 保留用户对参考图、对象名称、点击入口和显示时机的原始描述。
4. 用户追加修改说明时，需要追加到同一个步骤的原始表达中。

### 规范表达

`规范表达` 是编码前和修正过程中的执行依据。

要求：

1. 保留原始表达的有效意图。
2. 把 UI 对象、资源、布局、层级、显示隐藏、点击入口和主流程启动拆成独立规则。
3. 对资源路径进行实际存在性核对。
4. 如果原始表达路径与项目实际资源不一致，必须显式写出纠偏说明。
5. 明确哪些对象只是视觉元素，哪些对象是真实交互入口。
6. 明确 UI 显示期间主游戏流程是否启动。
7. 明确 UI 隐藏后主游戏流程从哪个阶段开始。

### 最终表达

`最终表达` 是实现成功并验证后的稳定沉淀。

要求：

1. 只记录最终确认采用的规则。
2. 记录最终采用的资源路径和实际文件格式。
3. 记录实际实现字段、关键方法、场景节点和脚本挂载关系。
4. 记录测试验证结果。
5. 能作为后续步骤的可靠前置基础。

## 入口界面阻塞规则

如果需求出现：

```text
游戏进入显示封面
点击某对象开始游戏
游戏开始前界面
```

规范表达必须明确：

1. 该界面是否在主流程之前显示。
2. 该界面显示时，主游戏对象是否隐藏。
3. 该界面显示时，主游戏逻辑是否暂停或尚未启动。
4. 点击开始后，主流程从哪个阶段或哪个方法启动。
5. 点击开始是否只响应一次。

示例：

```text
游戏脚本启动后，不立即执行 playOpeningDrop。
游戏脚本先显示封面，并隐藏 yu1-yu4 与 bg1-bg22。
玩家点击 fmlab01 或 fmlab02 后，封面隐藏，主游戏流程从开场掉落阶段开始执行。
```

## UI 对象定义规则

如果需求给出 UI 对象列表，规范表达必须转换成稳定对象表。

对象表必须包含：

1. 对象名称。
2. 资源路径。
3. 项目实际资源路径。
4. 对象用途。
5. 是否可点击。
6. 点击后行为。

如果原始表达中的资源路径不存在，规范表达不能静默采用。

必须写出：

```text
原始表达写作 A，项目实际资源为 B，实现时以 B 为准。
```

## 参考图规则

如果需求包含参考图，规范表达必须明确参考图的作用边界：

1. 参考整体视觉氛围。
2. 参考元素大致位置。
3. 参考层级关系。
4. 不把参考图当作自动切图来源。
5. 以用户提供的对象资源作为实际实现素材。

示例：

```text
封面布局、整体视觉氛围、按钮位置和元素层级参考 ck.png。
实际显示对象以 fmbk、fmlab01、fmlab02、fmstart、fmclose 对应资源为准。
```

## 视觉按钮与点击入口分离规则

如果需求中同时出现按钮资源和明确点击对象，规范表达必须区分：

1. 哪些对象只是视觉展示。
2. 哪些对象真实接收点击。
3. 视觉按钮是否一定等于点击入口。
4. 未被要求实现点击的按钮是否暂不绑定事件。

示例：

```text
玩家点击 fmlab01 时开始游戏。
玩家点击 fmlab02 时开始游戏。
fmstart 显示为视觉开始按钮，但本步骤不要求点击它开始游戏。
fmclose 显示在右上角，但本步骤不要求关闭功能。
```

## UI 层级规则

如果新增封面、弹窗、菜单或遮罩，规范表达必须明确：

1. UI 对象挂在哪个父节点下。
2. UI 对象之间的层级顺序。
3. UI 是否显示在主游戏对象之上。
4. UI 隐藏后是否还遮挡主游戏对象。
5. Cocos 场景中是否需要通过子节点顺序保证渲染层级。

示例：

```text
封面对象作为 Canvas 子节点存在。
封面节点位于 Canvas 子节点列表末尾，保证封面显示在主游戏对象之上。
封面隐藏后，不再遮挡主游戏对象。
```

## 显示隐藏规则

入口界面需求必须明确显示隐藏的对象集合：

1. 初始显示哪些 UI 对象。
2. 初始隐藏哪些游戏对象。
3. 点击后隐藏哪些 UI 对象。
4. 点击后显示哪些游戏对象。
5. 哪些对象仍由原有流程控制，不应立即显示。

示例：

```text
封面显示时，fmbk、fmlab01、fmlab02、fmstart、fmclose 全部显示。
封面显示时，yu1-yu4 和 bg1-bg22 隐藏。
点击开始后，封面对象全部隐藏，yu1-yu4 显示。
bg1-bg22 仍由原有 bg 掉落阶段控制显示。
```

## 脚本字段规则

新增 UI 管理逻辑时，规范表达必须要求脚本字段与场景序列化字段一致。

需要明确：

1. 脚本中新增的 `@property` 字段。
2. 场景中挂载到字段的节点。
3. 哪些节点属于统一显示隐藏集合。
4. 哪些节点属于点击入口集合。
5. 字段为空时是否允许自动跳过入口界面。

示例：

```text
coverNodes = [fmbk, fmlab01, fmlab02, fmstart, fmclose]
coverStartNodes = [fmlab01, fmlab02]
```

## 资源校验规则

涉及 UI Sprite 资源时，实现和最终表达必须检查：

1. 资源文件是否存在。
2. `.meta` 文件是否存在。
3. SpriteFrame UUID 是否引用正确。
4. 图片尺寸是否与布局规则一致。
5. 原始表达路径与实际路径是否存在差异。

最终表达必须记录最终采用的实际路径。

## 本地预览测试规则

封面、菜单、入口界面完成后，至少需要验证：

1. 首屏是否显示入口界面。
2. 主游戏对象是否没有提前显示。
3. 点击真实入口后是否进入主流程。
4. 点击入口是否只触发一次。
5. 控制台是否有 error。
6. 是否出现 missing script 或脚本挂载错误。

如果存在多个入口，必须分别验证。

示例：

```text
点击 fmlab01 后，封面隐藏并进入主游戏流程。
重新加载后点击 fmlab02，封面隐藏并进入主游戏流程。
```

## 验收标准生成规则

入口 UI 类验收标准按以下顺序组织：

1. 首屏显示规则。
2. 资源引用规则。
3. 布局位置规则。
4. 层级遮挡规则。
5. 初始隐藏和显示规则。
6. 点击入口规则。
7. 主流程启动规则。
8. 视觉按钮和真实入口分离规则。
9. 控制台、scene JSON、资源 UUID、脚本挂载校验。
10. 本地预览测试结果。

## 最终表达沉淀规则

从 `规范表达` 生成 `最终表达` 时，需要：

1. 写清最终采用的资源路径。
2. 写清原始资源路径与实际资源路径的差异。
3. 写清最终采用的节点层级。
4. 写清最终采用的点击入口。
5. 写清最终采用的脚本字段和关键方法。
6. 写清本地预览验证结果。
7. 避免保留“需要确认”“例如”等未完成语气。

## 写作风格

1. 使用明确、直接的中文。
2. 一个条目只表达一个判断点。
3. 对路径、节点名、方法名和字段名使用反引号。
4. 对 UI 入口、视觉元素、主流程启动分别成章描述。
5. 发现路径不一致时，必须显式写出纠偏结论。
