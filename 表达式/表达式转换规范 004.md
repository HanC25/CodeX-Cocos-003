# 表达式转换规范 004

## 目标

本规范基于 `表达式转换规范 003.md`，并吸收 `表达式 004` 系列在点击交互、命中区域、匹配反馈、通用消失规则和 Cocos Preview 测试中的经验。

用于指导后续把 `原始表达` 转换为 `规范表达`，再从实现结果沉淀为 `最终表达`。

本版本新增重点：

1. 交互需求必须明确“可操作对象”和“不可操作对象”。
2. 点击命中必须区分节点原点、UITransform 区域和可见内容区域。
3. 目标选择必须明确范围、筛选条件和最近距离口径。
4. 匹配结果必须明确是在点击时确定，还是反馈时重新计算。
5. 对象消失规则必须区分“是否触发反馈”和“是否消失”。
6. 本地预览测试必须确认加载的是最新脚本，不把旧 chunk 行为当作最新逻辑。

## 三类表达的职责

### 原始表达

`原始表达` 记录用户最初输入和后续追加的修改说明。

要求：

1. 保留用户原句。
2. 用户追加修改说明时，需要追加到同一个步骤的原始表达中。
3. 如果后续说明覆盖前面说明，原始表达可以同时保留旧说明和新说明。
4. 不在原始表达中主动删除用户曾经提出过的规则。

### 规范表达

`规范表达` 是编码前和修正过程中的执行依据。

要求：

1. 保留原始表达的有效意图。
2. 把模糊描述转为可执行规则。
3. 对交互对象、命中区域、目标选择、反馈触发和状态互斥进行明确拆分。
4. 对后续补充规则，补充到对应章节，而不是只追加在末尾。
5. 对实现风险明确写入验收标准。

### 最终表达

`最终表达` 是实现成功并验证后的稳定沉淀。

要求：

1. 只记录最终确认采用的规则。
2. 记录实际实现中修正过的问题。
3. 记录最终采用的代码字段、关键方法、场景字段和测试口径。
4. 能作为后续步骤的可靠前置基础。

## 可操作对象规则

如果需求出现：

```text
玩家可以操作 A
不可操作对象包括 B
点击 A 触发交互
```

规范表达必须明确：

1. 哪些对象可以接收玩家输入。
2. 哪些对象不能接收玩家输入。
3. 点击不可操作对象时是否完全无效。
4. 不可操作对象是否仍然可以被系统逻辑影响。
5. 交互入口绑定到对象本身，还是绑定到统一输入层。

示例：

```text
玩家只能点击 yu1-yu4。
bg1-bg22 不作为可点击对象。
点击 bg1-bg22 不触发匹配、不触发反馈，也不改变该 bg 的掉落状态。
```

## 点击命中规则

如果对象是 Spine、Sprite、带偏移的可见内容，或者前序步骤已经确认“按可见内容判断”，规范表达必须明确：

1. 点击命中按节点原点、UITransform 矩形、碰撞体，还是可见内容区域判断。
2. 如果节点原点与可见内容中心不一致，必须使用可见内容区域。
3. 如果使用 Canvas 统一接收点击，必须说明坐标转换方式。
4. 点击区域需要考虑对象缩放。
5. 验收标准必须包含“点击可见内容能命中”。

示例：

```text
点击事件由 Canvas 统一接收。
点击坐标转换为 1280x720 Canvas 本地坐标。
yu1-yu4 的命中区域按可见内容区域判断，而不是按节点原点判断。
```

## 目标选择规则

如果需求出现：

```text
距离最近的对象
同一条轴上最近的对象
匹配最近的 bg
```

规范表达必须明确：

1. 查找范围是什么。
2. 只查找可见对象，还是包含画布外对象。
3. 是否排除已经被其他交互锁定的对象。
4. 最近距离按 x、y、欧氏距离，还是轨道方向距离判断。
5. 判断点是节点原点、可见内容中心，还是其他自定义点。
6. 如果没有目标对象，本次点击如何处理。

示例：

```text
点击 yu 后，只在该 yu 对应的 z 轨道上查找当前画面内可见、仍在掉落、未被其他交互锁定的 bg。
最近距离以 yu 可见内容中心与 bg 节点中心点的 y 坐标差值绝对值判断。
```

## 匹配结果确定规则

如果交互包含“目标选择 -> 运动 -> 反馈”，规范表达必须明确：

1. 匹配结果是在点击时确定，还是反馈触发时确定。
2. 目标对象移动过程中是否重新选择最近对象。
3. 目标对象移动过程中是否重新计算匹配结果。
4. 被选中的目标对象是否还可以被其他对象选择。

示例：

```text
匹配结果在点击时确定。
目标 bg 快速下落过程中，不重新计算最近 bg，也不重新计算匹配结果。
被选中的目标 bg 不再参与其他 yu 的最近 bg 判断。
```

## 反馈与消失解耦规则

如果需求出现：

```text
无论是否有反馈，对象都消失
距离达到 N 时消失
正确或错误时反馈
```

规范表达必须把“消失”和“反馈”拆成两个规则。

必须明确：

1. 什么条件下对象消失。
2. 什么条件下触发反馈。
3. 对象消失但不触发反馈时，是否播放动画。
4. 被点击目标和非点击目标是否采用同一消失规则。
5. 消失后对象是否还能参与匹配和显示。

示例：

```text
无论是否存在反馈状态，只要 yu 与同 z 轨道 bg 的中心点垂直距离小于或等于 20px，该 bg 都立即消失。
如果该 bg 是点击选中的目标 bg，则同时触发正确或错误反馈。
如果该 bg 未被点击选中，则只消失，不触发 yu 的反馈动画。
```

## 反馈动画规则

如果需求包含动画序列，规范表达必须明确：

1. 动画名称。
2. 播放顺序。
3. 每个动画是否循环。
4. 序列结束后恢复到什么状态。
5. 动画缺失时是否允许静默失败。

示例：

```text
正确反馈播放 `5_chi -> 4_shengli -> 2_daiji`。
错误反馈播放 `5_chi -> 3_shibai -> 2_daiji`。
反馈结束后恢复步骤三中的浮动状态。
```

## 交互状态规则

交互需求必须明确状态互斥：

1. 同一个对象是否允许同时处理多个交互。
2. 点击后到反馈结束前是否锁定。
3. 反馈动画期间是否允许再次点击。
4. 不同对象之间是否互相影响。
5. 被选中的目标是否参与其他目标选择。

示例：

```text
同一时间，一个 yu 只能处理一个交互。
某个 yu 从点击命中目标 bg 开始，到反馈结束或目标 bg 离场前，不能再次触发新的匹配。
不同 yu 可以独立处理点击和反馈。
```

## Cocos 预览测试规则

使用 `localhost` 或 Cocos Creator Preview 测试时，必须注意：

1. 修改源码后，浏览器可能仍加载旧的 preview chunk。
2. 测试前需要确认 Cocos 已重新编译 `GameStart.ts`。
3. 如果本地链接没有表现出最新改动，先检查脚本 chunk 是否更新。
4. 不把旧脚本行为误判为新逻辑失败。
5. 必要时重新点击 Cocos 预览、刷新浏览器或重启预览服务。

最终表达需要记录本步骤是否经过用户本地验证。

## Cocos 场景字段规则

涉及新增或调整脚本属性时，规范表达必须要求：

1. `GameStart.ts` 的 `@property` 字段和 `main.scene` 序列化字段一致。
2. 新增字段必须能在 `main.scene` 中找到对应值。
3. `main.scene` 必须能 JSON 解析。
4. TypeScript 检查不能出现当前脚本文件相关错误。
5. 控制台不能出现 missing script 或脚本挂载错误。

## 验收标准生成规则

交互类验收标准按以下顺序组织：

1. 可操作对象和不可操作对象。
2. 点击命中区域。
3. 目标查找范围。
4. 最近目标选择规则。
5. 点击后的即时动作。
6. 目标对象运动规则。
7. 匹配结果确定规则。
8. 消失规则。
9. 正确反馈规则。
10. 错误反馈规则。
11. 状态互斥规则。
12. 控制台、TypeScript、scene JSON、脚本挂载校验。

## 最终表达沉淀规则

从 `规范表达` 生成 `最终表达` 时，需要：

1. 写清最终采用的所有数值。
2. 写清最终采用的点击命中口径。
3. 写清最终采用的目标选择口径。
4. 写清最终采用的消失与反馈关系。
5. 写清最终实现字段和关键方法。
6. 写清测试验证结果。
7. 避免保留“需要确认”“例如”等未完成语气。

## 写作风格

1. 使用明确、直接的中文。
2. 一个条目只表达一个判断点。
3. 对坐标、时间、距离、速度、随机范围使用数字和单位。
4. 对代码字段、动作名、资源路径使用代码格式。
5. 对消失、反馈、匹配、点击命中分别成段描述，不混写。
