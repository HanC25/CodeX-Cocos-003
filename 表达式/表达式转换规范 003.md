# 表达式转换规范 003

## 目标

本规范基于 `表达式转换规范 002.md`，并吸收 `表达式 003` 系列在 bg 循环掉落、多对象轨道调度、视觉反馈修正中的经验。

用于指导后续把 `原始表达` 转换为 `规范表达`，再从实现结果沉淀为 `最终表达`。

本版本新增重点：

1. 多对象循环生成需要明确“轮次”“开始掉落”“完成掉落”的定义。
2. 屏幕截图反馈需要反向修正规范表达。
3. 视觉间距规则必须明确是中心点间距、可见边缘间距，还是生成时间间隔。
4. 否定过的规则需要在规范表达和最终表达中明确“不采用”。
5. 最终表达必须记录最终采用的调度字段和验证口径。

## 三类表达的职责

### 原始表达

`原始表达` 记录用户最初输入和后续追加的修改说明。

要求：

1. 保留用户原句。
2. 用户追加修改说明时，需要追加到同一个步骤的原始表达中。
3. 如果后续说明覆盖前面说明，原始表达可以同时保留旧说明和新说明。
4. 不在原始表达中主动删除用户曾经提出过的规则。

### 规范表达

`规范表达` 是编码前和修正过程中的执行依据。

要求：

1. 保留原始表达的有效意图。
2. 把模糊描述转为可执行规则。
3. 对被后续否定的规则，明确写出“不采用”。
4. 对视觉反馈暴露的问题，补充更精确的验收标准。
5. 对多对象、多轨道、多轮次行为，明确调度方式。

### 最终表达

`最终表达` 是实现成功并验证后的稳定沉淀。

要求：

1. 只记录最终确认采用的规则。
2. 对曾经尝试但最终否定的规则，作为“非采用原则”说明。
3. 记录最终实现字段、资源路径、层级策略和验证结果。
4. 能作为后续步骤的可靠前置基础。

## 多对象轮次规则

如果需求出现：

```text
全部掉落一遍为一轮
掉落一轮后再掉落第二轮
如此往复
随机顺序
```

规范表达必须明确：

1. 一轮是“全部开始掉落一次”，还是“全部完成掉落一次”。
2. 每一轮是否重新随机顺序。
3. 当前轮结束后何时进入下一轮。
4. 是否允许多个对象同时在画面中存在。
5. 是否允许下一轮对象在上一轮对象仍可见时开始。

## 生成间隔规则

如果需求出现：

```text
前一个进入画面后 1 秒，下一个进入
间隔至少 5 秒
至少经过 N 秒
```

规范表达必须明确：

1. 间隔从哪个事件开始计算。
2. 下一个对象是“必须进入”还是“可以尝试进入”。
3. 如果轨道不可用，是否延后生成。
4. 时间单位必须写成秒。
5. 如果多个约束同时存在，需要说明优先级。

示例：

```text
前一个 bg 的可见内容全部进入画面后，再经过 1 秒，下一个 bg 可以尝试进入画面。
如果此时没有可用轨道，则延后生成，直到出现可用轨道。
```

## 轨道间距规则

涉及 z1-z4、同轨道、间隔、中心点距离时，必须明确：

1. 间距判断是 x 方向、y 方向，还是同一轨道方向。
2. 判断对象是节点原点、可见内容中心点，还是可见边缘。
3. 最小间距的单位。
4. 是否只在生成瞬间判断，还是持续保证。
5. 如果视觉截图显示不满足，需要修正规范而不是只解释。

示例：

```text
在 z1-z4 任意一条轨道方向上，同时正在下落的任意两个 bg，其中心点间隔至少为 260px。
该间隔以 bg 节点中心点的 y 坐标差值判断。
```

## 轨道冷却规则

如果需求要求：

```text
同一条轨道上，出现一个对象后，至少经过 N 秒，再出现下一个对象
```

规范表达必须明确：

1. 冷却绑定的是轨道，不是对象。
2. 冷却从该轨道生成对象的时刻开始计算。
3. 冷却未结束时，该轨道不可用。
4. 冷却结束后，该轨道仍需要满足其他空间间距规则。
5. 如果所有轨道都不可用，下一个对象延后生成。

示例：

```text
同一条 z 轨道生成一个 bg 后，需要设置 5 秒轨道冷却时间。
也就是说，同一条 z 轨道上出现一个 bg 后，至少经过 5 秒，才允许再出现下一个 bg。
```

## 否定规则沉淀

如果用户明确说：

```text
不是我希望的原则
不要这样
改为
```

规范表达必须做两件事：

1. 明确写出旧规则不采用。
2. 明确写出新规则作为执行口径。

最终表达中也需要保留非采用原则，避免后续再次误用。

示例：

```text
不采用“同一条 z 轨道上只要已有可见 bg，就不再生成新的同轨道 bg”的原则。
最终采用“同一条 z 轨道上出现一个 bg 后，至少经过 5 秒，再出现下一个 bg”。
```

## 资源路径规则

如果原始表达中的资源路径与项目实际路径不一致：

1. 原始表达保留用户原始路径。
2. 规范表达说明原始路径与实际路径差异。
3. 最终表达记录最终实现采用的实际路径。
4. 编码时必须使用项目中真实存在的资源路径。

示例：

```text
原始表达中资源路径写作 assets/art/sprites/bg*.png。
最终实现以项目实际资源路径 assets/art/common/sprites/bg*.png 为准。
```

## 层级规则

如果需求出现：

```text
A 和 B 重叠时，A 在前
A 不被 B 遮挡
```

规范表达必须明确：

1. 是渲染层级，不是碰撞关系。
2. 在 Cocos 中通过节点顺序、层级或其他明确方式实现。
3. 验收标准必须写“重叠时谁显示在前”。

## Cocos 场景字段规则

涉及新增或调整脚本属性时，规范表达必须要求：

1. `GameStart.ts` 的 `@property` 字段和 `main.scene` 序列化字段一致。
2. 删除不用的旧字段。
3. 新增字段必须能在 `main.scene` 中找到对应值。
4. `main.scene` 必须能 JSON 解析。
5. TypeScript 必须编译通过。

## 验收标准生成规则

验收标准按以下顺序组织：

1. 资源和节点数量。
2. 分类或数据关系。
3. 阶段启动条件。
4. 轮次和随机顺序。
5. 单个对象运动规则。
6. 多对象调度规则。
7. 轨道间距和时间冷却规则。
8. 层级显示规则。
9. 动画播放规则。
10. 控制台、TypeScript、scene JSON、资源 UUID 校验。

## 最终表达沉淀规则

从 `规范表达` 生成 `最终表达` 时，需要：

1. 写清最终采用的所有数值。
2. 写清最终采用的资源路径。
3. 写清最终采用的调度原则。
4. 写清最终不采用的原则。
5. 写清实现字段和验证结果。
6. 避免保留“可能”“需要确认”等未完成语气。

## 写作风格

1. 使用明确、直接的中文。
2. 一个条目只表达一个判断点。
3. 对坐标、时间、间距、速度、随机范围使用数字和单位。
4. 对代码字段、动作名、资源路径使用代码格式。
5. 对被否定的规则使用“不采用”明确表达。
